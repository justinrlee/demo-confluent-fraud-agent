# Running the Demo Applications

Quick reference for starting and managing the fraud detection demo services.

## Prerequisites

1. **Virtual environment setup:**
   ```bash
   python3 -m venv venv
   venv/bin/pip install -r requirements.txt
   ```

2. **Terraform deployed:**
   ```bash
   cd terraform && terraform apply
   ```
   This generates the `.env` file in the project root.

---

## Start Services

### Start All (Recommended)
```bash
./start-producer.sh        # Start event producer (default: normal transactions only)
./start-dashboard.sh       # Start Streamlit dashboard
./start-fraud-trigger.sh   # Start fraud trigger web UI
```

### Individual Services

**Producer (Event Generator):**
```bash
./start-producer.sh [MODE]
```
Modes:
- `(no args)` - Normal user activity only (default)
- `--fraud` - Fraud scenarios only
- `--both` - Normal + fraud after cycle 50
- `--single-fraud` - One fraud event then exit

**Dashboard:**
```bash
./start-dashboard.sh
```
Access at: http://localhost:8501

**Fraud Trigger UI:**
```bash
./start-fraud-trigger.sh
```
Access at: http://localhost:8080

## Stop Services

**Stop all services:**
```bash
./stop-all.sh
```

**Stop individual service:**
```bash
kill $(cat run/dashboard.pid)
kill $(cat run/fraud-trigger.pid)
kill $(cat run/producer.pid)
```

---

## Logs

All logs are written to the `run/` directory:

**View logs in real-time:**
```bash
tail -f run/producer.log
tail -f run/dashboard.log
tail -f run/fraud-trigger.log
```

**View recent activity:**
```bash
tail -n 50 run/producer.log
```

---

## Process Management

### PID Files
Each service stores its PID in `run/`:
- `run/producer.pid`
- `run/dashboard.pid`
- `run/fraud-trigger.pid`

### Check if a service is running:
```bash
ps -p $(cat run/producer.pid)
```

### Force kill if needed:
```bash
kill -9 $(cat run/producer.pid)
```

## Typical Workflow

### Demo Startup
```bash
# 1. Start producer (default: normal transactions only)
# Or use --both for mixed mode (normal + fraud after cycle 50)
./start-producer.sh

# 2. Start dashboard to visualize alerts
./start-dashboard.sh

# 3. Start fraud trigger for manual scenarios
./start-fraud-trigger.sh

# 4. Check everything is running
./status.sh
```

### During Demo
- **Dashboard:** http://localhost:8501 - Shows real-time fraud alerts
- **Fraud Trigger:** http://localhost:8080 - Click buttons to trigger specific fraud scenarios
- **Logs:** `tail -f run/producer.log` - Monitor event generation

### Demo Shutdown
```bash
./stop-all.sh
```

## Troubleshooting

### "Virtual environment not found"
```bash
python3 -m venv venv
venv/bin/pip install -r requirements.txt
```

### "Service is already running"
```bash
# Check if it's actually running
./status.sh

# Stop it first
kill $(cat run/dashboard.pid)

# Or stop all
./stop-all.sh
```

### "Connection refused" / Kafka errors
Check that `.env` file exists and contains valid credentials:
```bash
cat .env | grep BOOTSTRAP_SERVERS
```

Re-run Terraform if needed:
```bash
cd terraform && terraform apply
```

### Port already in use
- Dashboard uses port **8501** (Streamlit default)
- Fraud trigger uses port **8080** (configurable in start script)

Check what's using the port:
```bash
lsof -i :8501
lsof -i :8080
```

## Environment Variables

All three apps use a two-tier `.env` loading pattern:

1. **`~/.env`** (optional) - Global defaults across projects
2. **`.env`** (project root) - Project-specific config (takes precedence)

Generated by Terraform:
- `BOOTSTRAP_SERVERS`
- `KAFKA_API_KEY`
- `KAFKA_API_SECRET`
- `SCHEMA_REGISTRY_URL`
- `SCHEMA_REGISTRY_API_KEY`
- `SCHEMA_REGISTRY_API_SECRET`

## Files Overview

| File | Purpose |
|------|---------|
| `start-producer.sh` | Start event generator in background |
| `start-dashboard.sh` | Start Streamlit dashboard in background |
| `start-fraud-trigger.sh` | Start fraud trigger UI in background |
| `stop-all.sh` | Stop all running services |
| `status.sh` | Check service status and recent logs |
| `run/` | Directory for logs and PID files (auto-created) |
| `run/*.log` | Service logs |
| `run/*.pid` | Service PIDs |

## EC2 Deployment

On EC2, the same scripts work:

```bash
# SSH to instance
ssh -i key.pem ec2-user@<EC2-PUBLIC-IP>

# Setup (one time)
cd demo-confluent-fraud-agent
python3 -m venv venv
venv/bin/pip install -r requirements.txt

# Start services
./start-producer.sh        # Normal mode, or use --both for mixed
./start-dashboard.sh
./start-fraud-trigger.sh

# Check status
./status.sh
```

**Access from browser:**
- Dashboard: `http://<EC2-PUBLIC-IP>:8501`
- Fraud Trigger: `http://<EC2-PUBLIC-IP>:8080`

Ensure security group allows inbound TCP 8080 and 8501.
