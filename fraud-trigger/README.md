# Fraud Trigger Web Interface

Manual fraud scenario trigger for the Confluent Cloud fraud detection demo. Complements
the continuous producer (`producer/generate_events.py`) by providing on-demand fraud
event generation via a web UI.

## What This Is

A **FastAPI web application** with 6 buttons (3 fraud scenarios × 2 users) that sends
fraud events directly to Kafka topics. When you click a button:

1. Events are sent to `transactions`, `user_logins`, and/or `account_changes` topics
2. Flink windows the activity (3-second session per user)
3. The streaming agent (Confluent Intelligence + Claude) analyzes the profile
4. A fraud alert appears in the Streamlit dashboard (~30s delay for windowing)

**Use case:** During demos, trigger specific fraud patterns on-demand without waiting
for the continuous producer's random cycles.

---

## Available Fraud Scenarios

| Scenario | Events Sent | Pattern | Risk Score |
|----------|-------------|---------|------------|
| **Geo-Impossible** | 2 (login + transaction) | Login from Tokyo → Purchase in NYC seconds later | ~90 |
| **Account Takeover** | 3 (2 account changes + transaction) | Email change → Password change → Large purchase | ~95 |
| **Velocity** | 7 (login + 6 transactions) | 6 rapid transactions across different merchants | ~85 |

All scenarios target **user-49** or **user-50** (the designated fraud_target users
with low baseline spending, making fraud highly anomalous).

**Option B: Consume fraud_alerts Topic**

```bash
# From project root
./venv/bin/python - <<'PY'
import os, time
from dotenv import load_dotenv
from confluent_kafka import Consumer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroDeserializer
from confluent_kafka.serialization import MessageField, SerializationContext

load_dotenv(".env")
c = Consumer({
    "bootstrap.servers": os.environ["BOOTSTRAP_SERVERS"],
    "security.protocol": "SASL_SSL",
    "sasl.mechanisms": "PLAIN",
    "sasl.username": os.environ["KAFKA_API_KEY"],
    "sasl.password": os.environ["KAFKA_API_SECRET"],
    "group.id": f"verify-{time.time()}",
    "auto.offset.reset": "latest"
})
c.subscribe(["fraud_alerts"])
sr = SchemaRegistryClient({
    "url": os.environ["SCHEMA_REGISTRY_URL"],
    "basic.auth.user.info": f'{os.environ["SCHEMA_REGISTRY_API_KEY"]}:{os.environ["SCHEMA_REGISTRY_API_SECRET"]}'
})
d = AvroDeserializer(sr)

print("Listening for fraud alerts (trigger a scenario via web UI)...")
end = time.time() + 60
while time.time() < end:
    m = c.poll(1.0)
    if m and not m.error():
        v = d(m.value(), SerializationContext(m.topic(), MessageField.VALUE))
        if v["user_id"] in ["user-49", "user-50"]:
            print(f"✓ Alert: {v['user_id']} | risk={v['risk_score']} | {v['reasoning'][:80]}")
c.close()
PY
```

## API Reference

### `GET /`

Serves the HTML button UI.

**Response:** HTML page

---

### `POST /trigger`

Triggers a fraud scenario for a specific user.

**Request Body (JSON):**
```json
{
  "scenario": "geo_impossible" | "account_takeover" | "velocity",
  "user": "user-49" | "user-50"
}
```

**Success Response (200):**
```json
{
  "status": "success",
  "events_sent": 2,
  "scenario": "geo_impossible",
  "user": "user-49"
}
```

**Error Responses:**
- `400 Bad Request` - Invalid scenario or user
- `500 Internal Server Error` - Kafka delivery failure
- `504 Gateway Timeout` - Messages not acknowledged within 10s

**Example (curl):**
```bash
curl -X POST http://localhost:8080/trigger \
  -H "Content-Type: application/json" \
  -d '{"scenario":"velocity","user":"user-49"}'
```

## Architecture Details

### Persistent Producer

The Kafka producer is initialized **once on app startup** (via FastAPI lifespan event)
and shared across all requests. This amortizes the expensive Schema Registry connection
and schema fetches.

**Why not per-request?** Each producer initialization:
- Connects to Schema Registry
- Fetches 3 Avro schemas (transactions, user_logins, account_changes)
- Establishes Kafka broker connections

This adds 2-5 seconds per request. With a persistent producer, button clicks return in
~1-3 seconds (dominated by Kafka acknowledgment wait).

### Synchronous Response

The `/trigger` endpoint **blocks until Kafka acknowledges all messages** (via
`producer.flush(timeout=10)`). This ensures the user gets confirmation that the fraud
scenario was successfully sent.

**Trade-off:** 1-3s response time vs instant 202 Accepted with background processing.
For manual demo clicks, users expect feedback, so synchronous is better.

### Code Duplication

Fraud scenario generators (`_generate_geo_impossible`, `_generate_account_takeover`,
`_generate_velocity`) are **duplicated from `producer/generate_events.py`**, not
imported. This keeps the fraud-trigger app completely independent.

**Why?** The trigger app can evolve separately (e.g., add custom demo scenarios) without
risk of breaking the continuous producer.

### Credential Loading

All three applications (producer, dashboard, fraud-trigger) use a **two-tier `.env` loading pattern**:

```python
load_dotenv(os.path.expanduser("~/.env"))  # Optional defaults from home directory
load_dotenv(os.path.join(os.path.dirname(__file__), "..", ".env"), override=True)  # Project overrides
```

**Loading order:**
1. `~/.env` (home directory) - Optional global defaults (won't fail if missing)
2. `../.env` (project root) - Project-specific config with precedence (Terraform-generated)

This allows you to set common defaults in `~/.env` (e.g., for multiple projects) while
project-specific credentials in the project root take precedence.

## Troubleshooting

### "Connection refused" when accessing http://localhost:8080

**Fix:** Ensure uvicorn is running with `--host 0.0.0.0` (not `127.0.0.1`)

---

### "KeyError: BOOTSTRAP_SERVERS"

**Fix:** `.env` file missing or not in the correct directory. Copy from project root.

---

### Buttons show "Timeout: N messages not acknowledged"

**Fix:** Check Flink statements are RUNNING:
```bash
# From project root
cd terraform && terraform show -json | grep '"phase"'
```

DDL statements should be `COMPLETED`, continuous statements (`create-activity-profiles`,
`detect-fraud`) should be `RUNNING`.

---

### No fraud alerts appear in dashboard

**Possible causes:**
1. **Windowing delay:** Wait ~30s for session window to close and agent to process
2. **Dashboard offset:** Streamlit dashboard reads from `latest`, so it only shows alerts
   produced *after* it started. Restart the dashboard, then trigger fraud.
3. **Flink statements down:** Check Confluent Cloud UI or run `terraform show`

---

### "ERROR: Message delivery failed: Authentication failure"

**Fix:** Kafka API key/secret in `.env` is invalid or doesn't have ACL permissions.
Re-run `terraform apply` to regenerate credentials.

## Limitations & Notes

- **No TLS:** HTTP only; add ALB or nginx for HTTPS
- **No authentication:** Publicly accessible (fine for short demos)
- **No rate limiting:** Trust users not to spam
- **Hardcoded users:** Only user-49 and user-50 (matches fraud_target profile)
- **Session window timing:** Alerts appear 3–30s after click (depends on window firing)
